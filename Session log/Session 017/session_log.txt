WORK COMPLETED THIS SESSION
----------------------------

Started building a new top-level feature called "Habits" (originally proposed
as "Routines") — daily check-in routines authored by staff and assigned to
mentees. Lives in the "Offerings" group of the admin sidebar.

Design decisions locked in with the user before coding:
  - One-off habits are OUT of scope (skipped; always goes through a template).
  - Per-step checkboxes: a day is "successful" only when every step is checked.
  - goal_x_of_y window: auto-complete if goal met, auto-abandoned if window
    expires short.
  - Feature renamed from "Routines" to "Habits".
  - Notification/reminder delivery (email/SMS/push) is explicitly OUT of scope
    for Phase 1 — data model supports it but nothing is wired.

What actually shipped this session (the "metadata slice"):

1. Migration 954_habits_tables.sql — the full schema for all 5 tables:
   - habits                   template (name, desc, duration_mode, duration_days,
                               goal_successful_days, is_active, created_by)
     CHECK constraint validates the duration params against duration_mode.
   - habit_steps              ordered steps (title, instructions, order_index)
   - mentee_habits            assignment instance with snapshot fields
                               (name_snapshot, duration_mode_snapshot,
                               duration_days_snapshot, goal_successful_days_snapshot,
                               successful_days_count, start_date, end_date, status)
   - mentee_habit_steps       per-assignment snapshot of the steps
   - mentee_habit_step_logs   one row per (step, day) when mentee checks off
                              a step, with unique(mentee_habit_step_id, log_date)
   RLS: staff read in-org, admin/course_creator manage habit templates, any
   staff in org can manage mentee_habits, mentees can read/write their own
   step logs. Uses the same my_organization_id()/my_staff_role() helpers that
   998_offerings_tables.sql uses.

   NOTE: This migration file has NOT been applied to the Supabase instance
   yet — it just sits in the repo. It'll need to be applied manually (same
   as other migrations in this project) before the feature works end-to-end.

2. app/src/types/index.ts — added Habit, HabitStep, MenteeHabit, MenteeHabitStep,
   MenteeHabitStepLog, HabitDurationMode, MenteeHabitStatus.

3. app/src/lib/audit.ts — added 'habit' and 'mentee_habit' to EntityType union.

4. app/src/lib/habits.ts — NEW helper module with:
   - todayISO(), addDaysISO(), diffDays()      date helpers
   - durationSummary(d)                         human-readable summary
   - computeEndDate(startISO, d)                start+duration → end_date
   - successfulDatesFromLogs(logs, stepCount)   set of successful YYYY-MM-DD dates
   - successfulDayCount(...)                    count of successful days
   - computeStatus(mh, successfulDays, today)   decides active/completed/abandoned
   These are ready to power the mentee-side UI when it's built.

5. app/src/lib/modules.ts — added "Habits" entry to ALL_MODULES in the
   Offerings group (teal gradient, path /habits, letter H).

6. app/src/App.tsx — added 3 admin routes:
     /habits             → HabitsPage
     /habits/new         → HabitCreatePage  (admin | course_creator only)
     /habits/:id/edit    → HabitEditPage    (admin | course_creator only)

7. app/src/pages/HabitsPage.tsx — admin list view. Shows teal-icon rows with
   name, step count, duration summary, active/completed assignment counts.
   Toggle to show retired habits. "+ Create Habit" button visible to
   admin/course_creator. Uses supabaseRestGet for parallel fetch of habits,
   step counts, and assignment counts.

8. app/src/pages/HabitCreatePage.tsx — Zod + RHF form (matches Session 015
   validation pattern). Fields: name, description, duration_mode radio group,
   mode-specific inputs (duration_days and/or goal_successful_days). Zod
   superRefine validates duration params per mode. Writes audit log. On
   success, navigates to /habits/:id/edit.

9. app/src/pages/HabitEditPage.tsx — metadata-only edit form (step builder
   and assignment UI deliberately deferred, see OPEN ITEMS). Loads the habit,
   prefills the same RHF form, supports save / retire (toggle is_active) /
   delete. All mutations write audit log entries. Renders a prominent amber
   notice explaining that the step builder + mentee assignments are coming
   in the next session.
