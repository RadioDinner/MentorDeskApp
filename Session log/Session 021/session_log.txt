C) JOURNEYS FEATURE — FULL DESIGN + TODO LIST

   Mid-session the user pivoted to designing the next big feature:
   "Journeys". This is a flowchart-based way to sequence offerings
   (courses + engagements) into a decision-tree flow that a mentor
   can walk a mentee through. Think Zapier's flowchart editor.

   Concept:
     - FLOW (template)     = reusable flowchart, lives in
                             Offerings > Flows. Example: "General
                             Flow" starts with "Get to know us"
                             engagement, then decision node: course
                             OR waitlist, then 4x mentoring → 2x
                             → 1x → graduate.
     - JOURNEY (instance)  = per-mentee snapshot copy of a Flow
                             that can be edited independently for
                             that mentee.
     - Nodes: start, offering (course or engagement), decision,
              status (waitlist, graduated, etc.), end.
     - Connectors have LABELS — this is where the if/then/else
       logic lives (e.g. "if completed course", "if waitlist").
     - Mentor manually advances a Journey by clicking an outgoing
       connector from the current node.

   User decisions captured via AskUserQuestion:
     1. Scope: MVP this session → turned into "MVP next session",
        since the design itself ate the remaining time.
     2. Node semantics: NEW COMPANY SETTING —
        journey_auto_assign_offerings (default TRUE). When ON,
        entering an offering node auto-creates the mentee_offerings
        row. When OFF, the mentor gets a pending-assignment
        notification and manually confirms.
     3. Sidebar: new "Flows" module under the Offerings group.
     4. Legacy Mentee Flow: left alone in Company Settings, just
        fix the existing bug (see Item D below).

   RESEARCH FINDINGS (from two Explore agents):

   - THE BUG the user reported: OfferingEditPage.tsx has NO
     addToFlow field in its Zod schema (lines 35-53), NO checkbox
     in the UI, and NO save logic. OfferingCreatePage.tsx:57,168-175
     has it working correctly — so the fix is to mirror the create
     page pattern on the edit page.

   - CANVAS REUSE: 100% reusable infrastructure. lib/canvas.ts
     contains factories, constants, migrateContent, clientId. The
     drag-to-move / resize / undo / SVG connector machinery in
     CanvasEditPage.tsx is entirely type-agnostic. CanvasConnector
     already has a `label` field in the type (types:528) and
     createConnector already accepts a label — but labels are
     NOT currently rendered in the SVG. For Journeys, labels are
     critical, so the Journey flowchart editor either needs its
     own SVG render layer or we extend CanvasEditPage's.

   - GREENFIELD: no "next offering on completion" logic exists
     today. No triggers, no hooks. mentee_offerings has status =
     'active'/'completed'/'cancelled' with no cascade behavior.

   - EXISTING MENTEE FLOW: organizations.mentee_flow (jsonb) +
     mentees.flow_step_id (text). Purely a label/status system.
     MenteeFlowEditor lives in CompanySettingsPage.tsx:610-804.
     Migration 991 is the origin. Will be left as-is this session.
