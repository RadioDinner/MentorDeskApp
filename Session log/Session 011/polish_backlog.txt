3. [OPEN] Pagination + server-side filtering on list pages
─────────────────────────────────────────────────────────────
Zero pages currently paginate. Some pages (AuditLogPage) cap at
.limit(200) which is a ticking bug once any org has real data
volume. Others fetch all rows unbounded.

Why: at any non-trivial org scale, list pages will hang or crash
the browser. Payroll/Invoicing/Audit are most at risk.

Where to start:
  - AuditLogPage.tsx:129 — already has the 200 cap, easiest to
    retrofit with offset+limit pagination
  - InvoicingPage.tsx — largest list, highest frequency of use
  - PayrollPage.tsx, PairingsPage.tsx, MenteesListPage.tsx,
    PeopleListPage.tsx

Approach: Add a `usePaginatedList` hook (page, pageSize, total,
loading). Server-side via Supabase REST `Range` header + Content-
Range response. Keep a consistent PageBar component.

Scope: medium. The hook + component is a day, then each page is
a 10-minute migration.


─────────────────────────────────────────────────────────────
4. [OPEN] Unify error UX — toasts, skeletons, retry, empty states
─────────────────────────────────────────────────────────────
Current state: every page shows plain "Loading..." text, uses
inline red banners that persist until reload, no skeletons, no
toasts, no retry buttons (except inside LoadingErrorState).

Why: feels dated, confuses users, and makes errors feel worse
than they are. Success feedback also disappears because there's
nowhere for it to go.

Where to start:
  1. Add a ToastProvider context (app/src/context/ToastContext.tsx)
     with useToast() -> toast.success() / toast.error() / toast.info()
  2. Add a Skeleton primitive (components/ui/Skeleton.tsx)
  3. Replace inline banners in successful mutations with toasts
     (roughly 60+ sites — grep for setPaymentMsg, setError, set*Msg
     patterns)
  4. Replace "Loading..." text with Skeleton rows in list pages
  5. Wire errorReporter.ts into every mutation (currently only
     3 create pages)

Scope: 1-2 days for the provider + primitives, then incremental.


─────────────────────────────────────────────────────────────
5. [OPEN] Form validation pass — adopt Zod + React Hook Form
─────────────────────────────────────────────────────────────
Current state: HTML5 `required` only, no schema validation, no
field-level error messages. Error strings are guessed from
backend messages ("Email already exists" detected by substring
match in MenteeCreatePage.tsx:66).

Why: data integrity, better UX on create/edit flows, real
server-side validation rules.

Where to start:
  1. npm install zod react-hook-form @hookform/resolvers
  2. Create app/src/lib/schemas.ts with Zod schemas per entity
     (Mentee, Person, Offering, Pairing, Invoice, TimeCard)
  3. Migrate one create page as the template (MenteeCreatePage
     is smallest at 171 LOC)
  4. Propagate pattern to the other create/edit pages

Scope: template takes ~half a day; migration ~15 min per page
for ~12 pages.

10. [OPEN] Test scaffolding
─────────────────────────────────────────────────────────────
Current state: zero tests. No vitest.config, no test files.

Why: the biggest risk in the codebase. Anything pay/money/time
related can regress silently with financial consequences.

Where to start:
  1. npm install -D vitest @testing-library/react
     @testing-library/jest-dom jsdom
  2. Add vitest.config.ts
  3. Write pure-logic tests first (no DOM):
     - lib/scheduling.ts (complex block-picker logic)
     - lib/credits.ts (allocation math)
     - lib/dynamicFields.ts
     - lib/modules.ts
     - lib/format.ts (trivial but validates the new primitives)
     - salaryForPeriod() in PayrollPage.tsx (move to lib/ first)
  4. Add one integration test per critical list page
     (InvoicingPage status transitions, PayrollPage estimation)

Scope: half a day for scaffolding + format tests; then
incremental. Aim for 60% coverage on lib/ first.
