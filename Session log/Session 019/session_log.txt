1. STEP BUILDER  (app/src/pages/HabitEditPage.tsx)
   - Loads habit_steps for the habit on mount via supabaseRestGet.
   - "+ Add step" creates a new habit_steps row (via supabaseRestCall
     POST, same SDK-bypass pattern CourseBuilderPage uses for lessons)
     with order_index = current count, title = "Step N", then opens
     the inline editor.
   - Inline editor: title + optional instructions. "Save step" PATCHes
     the row, "Cancel" discards, "Delete" removes with re-indexing.
   - Drag-and-drop reorder — each <li> is draggable (unless in edit
     mode), with a shared stepDragRef / stepDragOver pair mirroring
     CourseBuilderPage's lesson reorder. On drop, we reindex locally
     and fire parallel PATCHes for order_index; on failure we roll the
     list back to the pre-drag snapshot and toast.
   - Every mutation writes a logAudit row with entity_type='habit' and
     details.sub = 'step_added' | 'step_updated' | 'step_deleted'.

2. MENTEE ASSIGNMENT UI  (app/src/pages/HabitEditPage.tsx)
   - "Mentee assignments" card below the step builder.
   - Assign form: mentee dropdown (filtered to active mentees in the
     org who don't already have an active assignment for this habit)
     + start_date picker defaulting to todayISO() + Assign button.
   - Assignment flow:
       a. computeEndDate() from lib/habits to derive end_date (null
          for until_x_successful).
       b. Insert mentee_habits row with all the snapshot columns
          (name_snapshot, description_snapshot, duration_*_snapshot,
          etc.) plus start_date, end_date, status='active',
          successful_days_count=0, assigned_by=profile.id.
       c. Bulk-insert mentee_habit_steps — one row per current
          habit_steps row, snapshotting title/instructions/order_index.
          Using supabaseRestCall with an array body (cast to
          Record<string, unknown>) since PostgREST accepts bulk insert
          but supabaseRestCall's body type is the single-row shape.
       d. Prepend to local assignments list, log audit row with
          entity_type='mentee_habit'.
   - Assignments display in two groups: Active and Past. Active rows
     have an "End" button that PATCHes status to 'abandoned' and sets
     completed_at.
   - Guard rails: cannot assign if habit has 0 steps, cannot assign if
     habit is retired, cannot assign to a mentee who already has an
     active assignment for this habit.

B) CANVASES v1.1 TIER-1 SCAFFOLDING  (uncommitted at wrap time —
   see commit shipped as part of wrap-up below)

   User asked for ideas beyond sticky notes and I proposed a tiered
   feature list. User picked "all of Tier 1":
     1. Resize notes
     2. Rich text inside a note (**bold**, *italic*, - bullets)
     3. Checklist note type
     4. Multi-select + bulk move
     5. Connectors / arrows between notes
     6. Keyboard shortcuts (N, Delete, Cmd+Z undo)
     7. Link notes (to courses, habits, canvases, or external URLs)

   Implementing all seven in one session proved to be ~1000 lines
   of new logic on top of CanvasEditPage. I started with the data-
   model scaffolding and the shared helpers, then the wrap was
   requested before the big CanvasEditPage rewrite landed. The
   scaffolding is valuable on its own and is staying on main so the
   rewrite can pick up cleanly tomorrow. What shipped this session:

   1. types/index.ts — CanvasNote is now a DISCRIMINATED UNION:
        CanvasStickyNote     { type: 'sticky', text }
        CanvasChecklistNote  { type: 'checklist', title, items[] }
        CanvasLinkNote       { type: 'link', label, linkType, linkId, linkUrl }
      All three share CanvasBaseNote (id, x, y, width, height,
      color, z). Added CanvasChecklistItem, CanvasConnector,
      CanvasLinkType, and extended CanvasContent to require
      `connectors: CanvasConnector[]`.

   2. lib/canvas.ts (NEW) — single source of truth for the new
      canvas model:
        - clientId(prefix): non-crypto id generator
        - COLORS array, colorClasses(c) helper
        - NOTE_DEFAULTS, NOTE_MIN, WORKSPACE_SIZE, HISTORY_LIMIT
          constants
        - createStickyNote / createChecklistNote / createLinkNote /
          createChecklistItem / createConnector factories
        - migrateContent(raw): normalizes a loaded content jsonb
          into the new shape. Legacy notes without a `type` field
          get upgraded to sticky notes. Unknown / malformed entries
          are dropped. Connectors array is defaulted to [].
        - renderMarkdown(text): tiny HTML-escape-first markdown
          renderer supporting **bold**, *italic*, and "- "/"* "
          bullet lists. Safe to feed into
          dangerouslySetInnerHTML. Three-feature scope by design.

   3. CanvasCreatePage: default content shape is now
      { notes: [], connectors: [] } so newly created canvases match
      the new type exactly.

   4. CanvasEditPage (stopgap — enough to compile, NOT the full
      rewrite):
        - Imports migrateContent from lib/canvas and runs the
          loaded canvas.content through it on mount, setting both
          `canvas` and `notes` from the normalized result.
        - addNote now emits `type: 'sticky'` so the literal
          satisfies the new union.
        - save PATCH builds content as { notes, connectors: (...) }
          instead of the old { notes } shape.
        - The render loop early-returns null for any non-sticky
          note. In practice there are no non-sticky notes (the
          editor UI for them ships in the rewrite), but this keeps
          the type narrowing clean.
        - The local COLORS / colorClasses / clientId / NOTE_DEFAULTS
          / WORKSPACE_SIZE duplicates at the top of CanvasEditPage
          are still there. They overlap with lib/canvas.ts but I
          deliberately left them to keep this checkpoint minimal;
          the rewrite will consolidate them.

   Build + tests after the stopgap: clean + 57/57.


DEFERRED TO SESSION 020 — CANVASES v1.1
----------------------------------------

The CanvasEditPage rewrite itself. Everything below is unbuilt and
is what the next session should pick up. Design notes are intentional
— they're the conclusions I reached while planning today before
running out of time, so session 020 can start coding instead of
re-deciding.

1. WORKSPACE DATA MODEL
   - Keep the two flat arrays: notes[] and connectors[].
   - Add local state for: selectedIds (Set<string>), editingNoteId
     (string | null), connectorMode (null | { sourceId: string | null }),
     and an undo history (past: CanvasContent[], cap at HISTORY_LIMIT).
     Redo stack was considered and tabled — V1 undo-only is fine.
   - History strategy: snapshot BEFORE any coarse action (add,
     delete, drag start, resize start, createConnector, deleteConnector,
     color change, starting a text edit). Do NOT snapshot on every
     keystroke.

2. RESIZE
   - Drag handle in the bottom-right corner of each note card.
   - Separate resizeStateRef from dragStateRef; both share the same
     document-level pointermove/pointerup effect.
   - Clamp to NOTE_MIN (140 x 100) and WORKSPACE_SIZE bounds.

3. RICH TEXT FOR STICKY NOTES
   - Display mode: render via dangerouslySetInnerHTML with
     renderMarkdown(note.text) (already implemented in lib/canvas.ts).
   - Edit mode: plain textarea on the markdown source.
   - Optional toolbar (B / I / •) that wraps the textarea selection
     in **, *, or prepends "- ". Pragmatic implementation: grab the
     textarea ref, compute new text, setNotes, then
     setTimeout(restoreSelection, 0). Finicky but workable. Can be
     skipped for V1 if time is tight — the raw markdown typing path
     works without the toolbar.

4. CHECKLIST NOTE TYPE
   - New "+ Checklist" button in the toolbar.
   - Renders a title input + a list of rows: [checkbox] [text input]
     [× delete]. Enter in a row creates a new row below.
   - "+ Add item" link at the bottom.
   - Toggling a checkbox updates items[i].done and pushes history.

5. LINK NOTE TYPE
   - New "+ Link" button.
   - Edit UI: type dropdown (course / habit / canvas / url) + value
     picker. For course/habit/canvas, lazy-load the relevant list
     (offerings where type=course, habits, canvases excluding self)
     the first time the picker is opened and cache in state. For
     url, a plain input.
   - Display: small card with an icon + label. Click opens the
     target (/courses/:id/edit, /habits/:id/edit, /canvases/:id,
     or external URL in a new tab).

6. MULTI-SELECT + BULK MOVE
   - Shift-click a note to add/remove from selectedIds.
   - Plain click = clear-then-select. Click empty canvas = clear.
   - When a drag starts on a selected note, all selected notes
     move together; dragStateRef needs a startPositions map keyed
     by noteId.
   - Selected notes get a visible ring (e.g. `ring-2 ring-brand`).

7. CONNECTORS
   - Toolbar "Connect" button enters connectorMode.
   - In connectorMode: first click on a note sets sourceId.
     Second click on a different note creates a connector with
     createConnector(...) and exits connectorMode. Click empty or
     press Escape to cancel.
   - Render as an SVG overlay positioned at 0,0 over the workspace.
     For each connector, draw a line from note center to note
     center, re-computed from current note state (so moving a note
     drags its connectors with it automatically).
   - Click a line to delete it (or show an inline × at the midpoint).

8. KEYBOARD SHORTCUTS + UNDO
   - Global keydown listener on document, scoped out of
     HTMLInputElement/HTMLTextAreaElement targets:
       N           — add a sticky note
       Delete/Backspace — delete selectedIds (if any)
       Cmd/Ctrl+Z  — undo (pop past, set current state)
   - Beware of onKeyDown bubbling from textareas — the scope check
     above is critical.

9. EXTRACT COMPONENTS IF IT HELPS READABILITY
   - CanvasEditPage is already ~540 lines. Adding all of the above
     in one file puts it around 1200+ lines, which is OK for this
     codebase (CourseBuilderPage is 728) but not great. Consider
     extracting:
       components/canvas/StickyNoteCard.tsx
       components/canvas/ChecklistNoteCard.tsx
       components/canvas/LinkNoteCard.tsx
       components/canvas/ConnectorLayer.tsx
     Each takes (note/connector, canEdit, isSelected, isEditing,
     handlers...) as props. Keep the top-level page focused on
     state + dispatch.
