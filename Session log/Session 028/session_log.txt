BUG FIXES
----------

  1. Reset Progress silently failed for mentors
     ROOT: migration 976 only granted staff SELECT on
     lesson_progress + question_responses; mentor DELETE was denied
     by RLS with no error bubbled up.
     FIX: migration 944 adds staff INSERT/UPDATE/DELETE policies for
     both tables. Also made resetCourse / resetLesson /
     markLessonComplete / removeCourse surface Supabase errors
     instead of swallowing them.

  2. Admins couldn't create automations
     ROOT: AutomationEditPage was doing a redundant staff lookup
     keyed on profile.user_id, which mis-resolved for admin users.
     FIX: profile IS the staff row — use profile.id directly as
     automation.owner_id.

  3. Audit log said "No audit entries found" for real admins
     ROOT: my_staff_role() used LIMIT 1 with no ORDER BY. Users
     with multiple staff rows (e.g. admin who created test mentor
     accounts via profile → My Accounts) could have the function
     return the non-admin row, blocking admin-scoped RLS.
     FIX: migration 940 rewrites my_staff_role() and
     my_organization_id() to prefer the highest-privilege role.

  4. Mentee saw their own lesson-completed notifications
     ROOT: in the multi-profile test setup both profiles share an
     auth.users.id; notifyUser was resolving the mentor's user_id
     and writing a notification to the same auth user.
     FIX: notifyUser now self-excludes if recipient_user_id ===
     current session user_id.

  5. Chunk load error after a deploy
     ROOT: Vite emits per-build hashed chunk filenames; a user's
     stale tab holds the old HTML that references filenames the CDN
     no longer serves.
     FIX: main.tsx + ErrorBoundary detect chunk load errors and
     reload the page once (sessionStorage loop guard).

DELIBERATE DESIGN DECISIONS
----------------------------

  * user_preferences is keyed on auth.users.id (not staff/mentee) so
    a user with multiple profiles sees consistent prefs across them.

  * notifications is keyed on auth.users.id for the same reason.

  * Automation engine lives in a Supabase edge function and runs with
    service role, so it can scan all automations in an org regardless
    of who owns them, and bypass RLS for writes. Clients invoke via
    supabase.functions.invoke, which carries the user's JWT.

  * 'manual' is a valid trigger type for automations that should fire
    only from journeys or direct-by-id invocation — not on global
    events.

  * Journey decision nodes now have TWO automation slots
    (reachAutomationId fires on arrival; automationId fires on
    completion) rather than one, so authors can notify-on-arrival AND
    also trigger side-effects when the task completes.

  * notifyUser is best-effort and never blocks the primary user
    action. Email/SMS stubs return without throwing.

  * Brand colors in Company Settings are deliberately disabled (UI
    is visible but greyed out) — not removed — so when brand theming
    is re-enabled, admins see their previously-saved values preserved.

TO DEPLOY FROM A FRESH SESSION
-------------------------------

  1. Apply pending migrations: `supabase db push`
  2. Deploy the edge function:
       supabase functions deploy process-automations
  3. git pull main
  4. cd app && npm install && npm run build
  5. Vercel auto-deploys from main.

Without step 1, Company Settings will error on save
("column default_course_completion_message not found"), and
mentor Reset Progress will continue silently failing.

Without step 2, automations save successfully in the UI but no fires
produce tasks or notifications.
