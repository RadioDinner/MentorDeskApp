B) BLOCK 14 — ADVANCE HANDLER + PENDING ASSIGNMENT  (commit 21290b8)

   This is the bulk of the mentee-side glue. The JourneyCard now
   renders one button per outgoing connector from the journey's
   current node, and clicking a button advances the journey. The
   handler branches on the target node type:

     end      → status='completed', completed_at=now
     offering → if journey_auto_assign_offerings is ON for the org,
                call assignOffering(targetNode.offeringId) to create
                the mentee_offerings row; if OFF, set
                pending_assignment_node_id so a mentor can manually
                confirm later. If the offering is already active on
                this mentee, skip both (no-op advance, no pending
                flag).
     decision → just advance (also clears stale pending flag)
     status   → just advance (also clears stale pending flag)

   assignOffering was lightly refactored:
     - Template lookup pulled from allOfferings state (was
       [...availableCourses, ...availableEngagements]) so it works
       even when the target offering is already filtered out of the
       pickers.
     - Returns Promise<string | null> — the inserted mentee_offering
       id on success, null on failure — so advanceJourney can detect
       success and decide whether to actually advance the journey.
     - All of the existing side effects (setup-fee invoice auto-
       create, engagement auto_send_invoice, mentee_flow step sync)
       still fire, so journey auto-advance has full parity with a
       manual Assign click.

   New helper confirmPendingAssignment(journey):
     - Called from the JourneyCard's amber "Confirm" button.
     - Calls assignOffering unless the offering is already active.
     - Clears pending_assignment_node_id on the journey row.
     - Audits mentee_journey/confirmed_assignment.

   JourneyCard rebuilt:
     - Outgoing connector buttons, color-tinted by target type
       (rose=end, violet=offering, amber=decision, gray=status),
       labeled with conn.label or the target node label as fallback,
       with a truncating arrow chip.
     - Amber "Pending assignment" callout with offering name and
       Confirm button when pending_assignment_node_id is set.
     - "No outgoing connectors" italic notice for dead-end journeys
       where the author forgot to wire a path forward.
     - New busy prop disables all buttons while an advance is in
       flight (shares the existing `assigning` state).
     - New nodeLabel() inline helper replaces the old currentLabel
       IIFE so the same lookup can be reused for target nodes in the
       connector button labels.

   New migration 948_mentee_journey_pending_assignment.sql adding
   pending_assignment_node_id text column to mentee_journeys. Per
   CLAUDE.md rule #6 numbering counts down — previous lowest was
   949_journey_auto_assign.sql.

   Type update: MenteeJourney gained pending_assignment_node_id:
   string | null.

   MenteesListPage pending-journey badge (slice 4 of Block 14):

NOTES FOR SESSION 025
---------------------

The Journeys feature is now end-to-end functional for any org with
journey_auto_assign_offerings = true. A mentor can:

  1. Author a flow (/flows, Session 022/023 work)
  2. Assign the flow to a mentee (Manage → Journeys → Start)
  3. Advance the mentee along the flow with one click per step
  4. See real mentee_offerings rows get created when the journey
     passes through offering nodes
  5. Watch the journey auto-complete when it reaches an end node

For orgs with auto-assign OFF, the advance flow also works — it just
leaves a pending_assignment flag that surfaces as an amber badge on
MenteesListPage and a Confirm button on the JourneyCard itself. Those
orgs still need Block 15 (the UI toggle to flip the setting); until
then the toggle can only be flipped via direct SQL.

Before running Block 15, run `npm ci` in app/ if node_modules is
missing. CompanySettingsPage is already a big file (~32 kB chunk) and
the toggle should land in whatever tab groups the other org-level
settings. A grep for `allow_multi_engagement` will find the right
place quickly.

Block 14 introduced a type-level concern worth noting: advanceJourney
narrowing of JourneyNode relies on the discriminated union staying
tight. If a new node type is added in journeyFlow types, both the
advance handler and the JourneyCard's nodeLabel() helper need to
handle it — they currently default to node.label || '(unlabeled)'
which is correct for label-carrying variants but would silently drop
a new typed variant.

One thing NOT tested end-to-end this session: a journey reaching an
end node, auto-completing, and the JourneyCard correctly sliding into
the completed list via the local optimistic state update. The logic
looks right (status + completed_at + activeJourneys.filter +
completedJourneys.filter) but there's no integration test. Worth
smoke testing in Block 16.
